# ТЗ — Дайджест-бот чата со ссылками

Документ из двух частей:
- **Часть 1 — для человека:** суть, зачем, что получится, ключевые решения.
- **Часть 2 — для агента-исполнителя:** постановка, которую можно отдать
  ИИ-агенту и получить рабочую фичу.

---

# Часть 1. Для человека

## Суть
Бот делает дайджест Telegram-чата: по каждому топику форума — несколько пунктов
с **сутью обсуждения** (что решили / получили / какой вопрос открыт) и **прямой
ссылкой** на ключевое сообщение. Не «35 сообщений про VPN», а смысл + переход
в один тап.

## Зачем
Закрывает сквозную боль больших чатов: участники тонут в потоке и не находят, что
уже обсуждали; стандартные боты-саммаризаторы путают треды и дают «воду».
Конкурент существующему дайджест-боту, но глубже и точнее.

## Что получится
```
⭐ Кейсы и результаты
• Оркестрация под Okdesk: 15 мин руками → 5 мин      ← кликабельно, в сообщение
• Авто-диагност для Volkswagen Touareg (#шаг6)
💬 Чат общения
• Anthropic: +8× кода на инженера (с оговоркой про метрику)
```
Запуск одной командой/по крону; отправка в нужный чат или в личку — с уже
раскрытой разметкой и живыми ссылками.

## Ключевые решения (почему так)
1. **Ссылки строит код, не LLM** — поэтому битых/выдуманных ссылок не бывает.
2. **LLM возвращает данные (JSON), а не готовый текст** — экранирование и формат
   не его забота, ломаться нечему.
3. **Топик — first-class** (настоящие имена + корректная группировка) — заголовки
   совпадают с чатом, ссылки ведут в нужный топик.
4. **Доставка в HTML через API** — разметка раскрывается всегда.
5. **Бережный доступ + согласие** — без флуд-банов и приватных рисков.

## Объём
- **В scope:** дайджест за период (по умолчанию 24ч), отправка, CLI + крон.
- **Вне scope:** дашборд, поиск по архиву (есть в Chat Intel отдельно),
  модерация контента.

# Часть 2. ТЗ для агента-исполнителя

## 0. Миссия
Реализовать фичу `digest` в существующем проекте **Chat Intel**
(`~/Projects/active/chat-intel`): из распарсенного чата собрать дайджест по
топикам со ссылками и отправить его. Переиспользовать инфраструктуру проекта,
не дублировать.

## 1. Окружение и переиспользование
- Стек: Python 3.12, Telethon (user-сессия, MTProto), SQLite, faster-whisper,
  headless `claude -p --output-format json` как LLM (не API-ключ).
- Переиспользовать:
  - `core/config.py` → `Config` (api_id/api_hash/session_path/db_path/target_chat…).
  - `ingest/telegram.py` → `_make_client(cfg)` (бережный клиент).
  - `core/db.py` → подключение к SQLite (WAL, busy_timeout).
- Новый код — в пакете `digest/`. CLI — подкоманда в `run_cycle.py`.

## 2. Инварианты (НЕ нарушать — это архитектура, а не пожелания)
1. **Ссылки детерминированы.** Их строит только код из `(chat_id, topic_id,
   msg_id)`. LLM не выдаёт ссылок.
2. **Топик first-class.** `topic_id` (корень форум-топика) хранится в БД; имена
   топиков берутся из `GetForumTopics`, не угадываются.
3. **LLM возвращает строгий JSON** (суть + `msg_id`); ссылки и разметку строит код.
4. **Доставка — HTML через API** с `parse_mode`.
5. **Бережный доступ:** одна Telethon-сессия, без параллельных команд (общий
   session-файл), мягкие ретраи. Дайджест — только для разрешённого чата.

## 3. Модель данных
Добавить в `messages` колонку `topic_id INTEGER` (nullable; NULL = не форум/General).
- Миграция: `ALTER TABLE messages ADD COLUMN topic_id INTEGER;` (идемпотентно).
- Заполнять при ингесте:
```python
def topic_root(msg) -> int | None:
    r = msg.reply_to
    if r and getattr(r, "forum_topic", False):
        return r.reply_to_top_id or r.reply_to_msg_id   # корень топика, не родитель
    return None
```
- Бэкфилл существующих строк: где возможно — по `reply_to_tg_id` верхнего уровня;
  где неоднозначно — оставить NULL (на дайджест за свежий период не влияет).

## 4. Компоненты (модули и сигнатуры)

`digest/topics.py`
```python
async def fetch_topic_titles(client, chat_id: int) -> dict[int, str]:
    """GetForumTopicsRequest → {topic_id: title}. Кэшировать на запуск."""
```

`digest/select.py`
```python
def fetch_window(conn, chat_id: int, since: str) -> list[dict]:
    """Сообщения за период (since — напр. '-1 day'), text<>'' , с полями
    id, sender, date, topic_id. Сгруппировать по topic_id."""
```
SQL:
```sql
SELECT tg_message_id, sender_name, date, text, topic_id
FROM messages
WHERE chat_id = :cid AND date >= datetime('now', :since) AND text <> ''
ORDER BY topic_id, date;
```

`digest/summarize.py`
```python
def build_input(groups, titles) -> dict   # → {"topics":[{topic_id,title,messages:[{id,sender,text}]}]}
def summarize(payload: dict) -> dict       # claude -p (Промпт B) → валидный JSON
```
- Вызов LLM: prompt(Промпт B) + JSON входа → stdin → `claude -p --output-format json`.
- **Валидация ответа:** соответствие схеме; **каждый `msg_id` обязан существовать
  во входе** — несуществующие пункты выкинуть. Невалидный JSON → повтор до 3 раз,
  затем поднять ошибку.

`digest/render.py`
```python
def msg_link(chat_id: int, topic_id: int | None, msg_id: int) -> str:
    short = str(chat_id).removeprefix("-100")
    return (f"https://t.me/c/{short}/{topic_id}/{msg_id}" if topic_id
            else f"https://t.me/c/{short}/{msg_id}")

def render_html(dig: dict, chat_id: int) -> str:
    import html
    out = []
    for t in dig["topics"]:
        out.append(f"<b>{html.escape(t['title'])}</b>")
        for b in t["bullets"]:
            url = msg_link(chat_id, t["topic_id"], b["msg_id"])
            out.append(f'• <a href="{url}">{html.escape(b["summary"])}</a>')
        out.append("")
    return "\n".join(out).strip()
```

`digest/deliver.py`
```python
async def deliver(client, target, text: str) -> int:

    target — username / id чата или личка владельца."""
```

## 5. CLI
Подкоманда в `run_cycle.py`:
```
run_cycle.py digest --chat <chat_id> --since "1 day" --to <target> [--dry-run]
```
- `--dry-run` — печатает HTML в stdout, не отправляет.
- Крон/launchd — опционально (по аналогии с `cycle`), один `flock`.

## 6. Контракты данных
Вход в LLM:
```json
{ "topics": [ { "topic_id": 487, "title": "Кейсы и результаты",
  "messages": [ {"id": 12804, "sender": "Денис", "text": "..."} ] } ] }
```
Выход LLM (строго):
```json
{ "topics": [ { "topic_id": 487, "title": "Кейсы и результаты",
  "bullets": [ {"summary": "Оркестрация под Okdesk: 15 мин → 5 мин",
               "msg_id": 12804} ] } ] }
```

## 7. Промпты

### Промпт B — выжимка в JSON (основной)
```
Ты — редактор дайджеста чата. На входе сообщения за период, сгруппированные по
топикам форума. Верни СТРОГО JSON — только данные, без текста вокруг.

ПРАВИЛА:
1. Пиши СУТЬ, не статистику. Запрещено «N сообщений про X». Каждый пункт — что
   решили / что получилось (с цифрой, если есть) / какой вопрос открыт / инсайт.
2. 1–5 пунктов на топик, только значимое. Приветствия, реакции, оффтоп — выкинь.
3. Для каждого пункта укажи "msg_id" — id РОВНО ОДНОГО самого показательного
   сообщения из входа. Бери id только из входа, не выдумывай. Ссылки и
   форматирование НЕ пиши — это сделает код.
4. "topic_id" скопируй из входа. Пустой топик не включай.
5. Инфостиль: по-русски, плотно, без воды и канцелярита.

ВХОД:
{ "topics": [ { "topic_id": 487, "title": "...",
  "messages": [ {"id": 123, "sender": "...", "text": "..."} ] } ] }

ВЫХОД (строго такой JSON, без markdown-обёртки):
{ "topics": [ { "topic_id": 487, "title": "...",
  "bullets": [ {"summary": "...", "msg_id": 123} ] } ] }
```

### Промпт A — анализ корпуса (опционально, для разбора тем/болей)
Корпус режется на куски ~2000 сообщений, на каждый — отдельный агент, потом синтез.
Строка входа: `YYYY-MM-DD HH:MM ⟫ sender ⟫ text`.
```
Проанализируй ВСЕ сообщения этого куска чата и верни СТРУКТУРИРОВАННЫЙ отчёт
на русском по разделам:
1. Период. 2. О чём беседы (8–12 тем + частота). 3. Боли и блокеры (с цитатами,
раздели техн./обучение/организац.). 4. Частые вопросы. 5. Жалобы/пожелания к
продукту. 6. Динамика и тон.
Только по фактам из сообщений. Кратко. Весь ответ — это и есть отчёт.
```

## 8. Требования к надёжности (обязательны)
- **Ссылка ведёт в нужное сообщение нужного топика** (через `msg_link` +
  `topic_id`; ручная проверка ≥5 ссылок при приёмке).
- **Имена топиков совпадают с чатом** (из `GetForumTopics`).
- **Никаких ссылок/разметки от LLM** (лишние поля игнорируются, несуществующие
  `msg_id` отбрасываются).
- **Разметка раскрывается** (HTML + `parse_mode`).
- **Нет флуд-бана** при штатном запуске (бережный клиент, без параллели).
- **Идемпотентность отправки** (опц.): не слать повторно тот же период.

## 9. Тесты (минимум)
- `topic_root()` — форум / не форум / ответ-на-ответ.
- `msg_link()` — с топиком и без, корректное снятие `-100`.
- `render_html()` — экранирование `& < >`, корректные `<a href>`.
- `fetch_window()` — фильтр пустых, группировка по топику, граница периода.
- `summarize()` — мок LLM: валидный JSON проходит; невалидный → ретрай; пункт с
  несуществующим `msg_id` отбрасывается.
- e2e на фикстуре: вход-сообщения → ожидаемый HTML.

## 10. Definition of Done
- `run_cycle.py digest --chat <id> --since "1 day" --to <target>` выдаёт дайджест,
  отправляет с живыми ссылками и реальными именами топиков.
- `--dry-run` печатает HTML.
- Все тесты зелёные; код проходит линт проекта.
- Прогон на реальном чате (`-1003399225532`): 5 случайных ссылок открывают
  верные сообщения; заголовки совпадают с топиками.
- Обновлена `[[Chat Intel — архитектура]]` (новый модуль `digest/`).

## 11. Декомпозиция (порядок реализации)
1. Миграция `topic_id` + заполнение при ингесте (`topic_root`) + бэкфилл.
2. `digest/topics.py` — кэш имён топиков.
3. `digest/select.py` — выборка за период.
4. `digest/summarize.py` — вход + `claude -p` (Промпт B) + валидация/ретраи.
5. `digest/render.py` — `msg_link` + HTML.
6. `digest/deliver.py` + CLI `digest` (+ `--dry-run`).
7. Тесты, прогон на реальном чате, обновление архитектурной заметки.
