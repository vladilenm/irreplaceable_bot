# Quick Task 260501-g11: Add `environment:` Block to docker-compose.yml Summary

**One-liner:** Explicit `environment:` block in docker-compose.yml with `${VAR}` interpolation for all 15 env vars â€” unblocks Timeweb compose sanitizer so dashboard-defined env vars reach the container at runtime.

## Problem

Production deploy on Timeweb App Platform was crash-looping with:

```
Missing required environment variable: BOT_TOKEN
    file:///app/dist/config.js:4
```

Root cause: Timeweb's compose sanitizer only forwards dashboard-defined env vars into the container when each var is **explicitly** listed in the service's `environment:` block via `${VAR}` interpolation. Without the block, dashboard values are dropped before `process.env` is populated, and `requireEnv('BOT_TOKEN')` in src/config.ts:4 fail-fasts at boot.

## Changes

### docker-compose.yml

Added one comment paragraph and one `environment:` block under `services.bot`. No other fields touched.

**Required env vars (5)** â€” bare `${VAR}` interpolation, fail-fast if unset:

| Var                        | Source in src/config.ts |
| -------------------------- | ----------------------- |
| `BOT_TOKEN`                | `requireEnv`            |
| `TARGET_CHAT_ID`           | `requireEnvInt`         |
| `AI_RADAR_THREAD_ID`       | `requireEnvInt`         |
| `AI_API_KEY`               | `requireEnv`            |
| `THREAD_SUMMARY_THREAD_ID` | `requireEnvInt`         |

**Optional env vars (10)** â€” `${VAR:-default}` form, defaults mirror src/config.ts:

Every value is double-quoted to keep YAML unambiguous around the `${VAR:-default}` colon.

## Why explicit declaration (Timeweb sanitizer note)

Future deploy diagnostics: if a NEW env var is added to src/config.ts, it MUST also be added to this `environment:` block. Otherwise the Timeweb dashboard value will be silently dropped at deploy time, exactly mirroring this incident. The `environment:` block is the contract between the Timeweb dashboard UI and `process.env` inside the container.

For local dev, `docker-compose.override.yml` (gitignored) supplies `env_file: .env` and the `./data:/app/data` volume â€” Compose v2 also reads `.env` automatically next to `docker-compose.yml` for `${VAR}` interpolation, so the new block works locally without modifying the override file.
