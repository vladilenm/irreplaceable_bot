<objective>
Make `docker-compose.yml` deployable on Timeweb App Platform (PaaS) without breaking local-dev semantics.

Purpose:
- Timeweb App Platform injects env vars into the container directly through its UI dashboard — there is no `.env` file on the host. The current `env_file: - .env` causes the platform build/run to fail because the file does not exist in the platform's build context.
- Bind-mount `./data:/app/data` requires a writable host directory at a known path. On Timeweb's managed runtime there is no persistent host filesystem outside attached Persistent Disks; bind mounts to relative host paths get wiped on redeploy. A Docker **named volume** is the portable primitive that both Docker Engine (local) and Timeweb's PaaS understand — on Timeweb the operator attaches a Persistent Disk to `/app/data` through the UI, which functionally replaces the named volume at runtime; locally, Docker Engine manages `botdata` and it survives `docker compose down` (only `down -v` or explicit `volume rm` destroys it).

Output: A single edited `docker-compose.yml` that drops `env_file`, swaps the bind-mount for a named volume `botdata`, declares the top-level `volumes:` block, keeps `restart: unless-stopped` + `logging` untouched, and rewrites the SETUP-06 comment to document the new persistence model.
</objective>
