# Quick 260428-n29: LLM raw-response diagnostic log Summary

**One-liner:** Added info-level `logger.info({ rawResponseHead, rawResponseLength, model }, 'AI raw response (debug)')` in `filterArticles` so the curator/digest pipeline records the actual LLM output (first 800 chars + full length) at production log level — diagnoses downstream parser/formatter regressions without enabling LOG_LEVEL=debug.

## Objective

Curator/digest pipeline previously logged only metadata (`'AI filtering complete'` with `model` + `inputArticles`); when downstream parsing or formatting misbehaved (e.g. recent arrow-format / quota regressions), there was no record of what the LLM actually returned. A bounded head + length lets us diagnose without dumping multi-KB payloads.

In `src/services/ai.service.ts`, immediately after the if/else block where `result: string` is assigned in both Claude (Anthropic SDK) and OpenAI branches and BEFORE the existing `'AI filtering complete'` log:

## Decisions Made

- **Log level = info, not debug:** the explicit goal is to inspect production behaviour without flipping LOG_LEVEL; debug would defeat the purpose.
- **Slice = 800 chars, hardcoded:** plan specified exact number; no config knob added (avoids surface area for a diagnostic that may be removed later).
- **Field names camelCase:** mirrors existing `inputArticles` field on adjacent log call — pino object convention in this file.
- **Placed BEFORE 'AI filtering complete':** keeps the two info logs adjacent so a single grep window in operational logs surfaces both metadata and content for the same call.
